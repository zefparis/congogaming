# PROMPT WINDSURF — MODULE JEU RESPONSABLE CONGO GAMING

## CONTEXTE
Stack : NestJS backend (Railway), Next.js frontend (Vercel), PostgreSQL/Prisma, Supabase.
Dashboard admin existant : congogaming.com/admin
Table joueurs existante avec champs : phone, kyc_status, risk_level, balance, blocked.

## CE QU'ON CONSTRUIT

Un module complet de jeu responsable en 2 parties :
1. **Côté Admin** — notes internes + historique + envoi d'avertissement
2. **Côté Joueur** — parcours progressif bienveillant avec auto-évaluation et auto-exclusion

## PARTIE 1 — BACKEND (NestJS + Prisma)

### 1.1 Migrations Prisma — ajouter ces champs/tables

```prisma
// Sur le modèle Player existant, ajouter :
model Player {
  // ... champs existants ...
  risk_note        String?          // Note interne admin
  risk_note_date   DateTime?
  risk_note_author String?
  warning_sent_at  DateTime?        // Date dernier avertissement envoyé
  self_excluded    Boolean          @default(false)
  self_exclusion_until DateTime?    // null = définitif
  rg_questionnaire_done Boolean     @default(false)
  rg_questionnaire_date DateTime?
  rg_answers       Json?            // Stocke les réponses au questionnaire
  block_reason     String?          // Raison visible joueur
  playerActions    PlayerAction[]
}

// Nouveau modèle historique actions
model PlayerAction {
  id         String   @id @default(cuid())
  playerId   String
  player     Player   @relation(fields: [playerId], references: [id])
  type       String   // "WARNING_SENT" | "BLOCKED" | "UNBLOCKED" | "NOTE_ADDED" | "SELF_EXCLUDED" | "RG_QUESTIONNAIRE"
  note       String?
  author     String?  // "ADMIN" ou phone joueur
  createdAt  DateTime @default(now())
}
```

### 1.2 Nouveaux endpoints NestJS

**PATCH /admin/players/:id/note**
- Body : `{ note: string, author: string }`
- Action : met à jour risk_note, risk_note_date, risk_note_author
- Crée un PlayerAction type "NOTE_ADDED"

**POST /admin/players/:id/warning**
- Body : `{ message: string, author: string }`
- Action : met à jour warning_sent_at, enregistre le message
- Crée un PlayerAction type "WARNING_SENT"

**PATCH /admin/players/:id/block**
- Body : `{ reason: string, author: string }` — reason visible par le joueur
- Action : blocked = true, block_reason = reason
- Crée un PlayerAction type "BLOCKED"

**PATCH /admin/players/:id/unblock**
- Body : `{ author: string }`
- Action : blocked = false, block_reason = null
- Crée un PlayerAction type "UNBLOCKED"

**GET /admin/players/:id/actions**
- Retourne l'historique PlayerAction trié par date DESC

**POST /players/:id/self-exclude**
- Body : `{ duration: "7d" | "30d" | "permanent" }`
- Auth : joueur authentifié uniquement
- Action : self_excluded = true, self_exclusion_until = date calculée ou null si permanent
- Crée un PlayerAction type "SELF_EXCLUDED"

**POST /players/:id/rg-questionnaire**
- Body : `{ answers: { q1: boolean, q2: boolean, q3: boolean } }`
- Auth : joueur authentifié
- Action : rg_questionnaire_done = true, rg_questionnaire_date = now(), rg_answers = answers
- Si 2+ réponses à risque → risk_level = "HIGH", envoie notification admin
- Crée un PlayerAction type "RG_QUESTIONNAIRE"

**POST /players/:id/reactivate**
- Auth : joueur authentifié
- Condition : self_exclusion_until < now() ET rg_questionnaire_done récent (< 24h)
- Action : self_excluded = false, blocked = false
- Crée un PlayerAction type "UNBLOCKED"

### 3.3 QUESTIONNAIRE JEU RESPONSABLE — 3 questions

Design : card centrée, ton chaleureux, pas agressif.
Titre : "Quelques questions pour jouer sereinement"

**Question 1 :**
"Je joue principalement pour le plaisir et le divertissement"
→ [ Oui, tout à fait ] [ Pas vraiment ]

**Question 2 :**
"Mon budget jeu est prévu à l'avance et ne dépasse pas ce que je peux me permettre"
→ [ Oui, toujours ] [ Pas toujours ]

**Question 3 :**
"Une personne proche ne s'est jamais inquiétée de mon rapport au jeu"
→ [ C'est le cas ] [ Cela m'est arrivé ]

**Logique de résultat :**
- 0 réponse à risque → message positif + bouton "Continuer à jouer"
- 1 réponse à risque → message bienveillant + "Continuer" + "Me fixer une limite"
- 2-3 réponses à risque → message de soutien fort + bouton "M'auto-exclure temporairement" mis en avant + "Continuer quand même" en secondaire discret

**Message 0 risque :**
"Parfait ! Vous avez une approche saine du jeu. Profitez bien !"

**Message 1 risque :**
"Merci pour votre honnêteté. Pensez à vous fixer un budget quotidien. Congo Gaming est là pour le plaisir, pas pour le stress."

**Message 2-3 risques :**
"Nous vous remercions de votre confiance. Votre bien-être passe avant tout. Nous vous recommandons de faire une pause. Vous pouvez revenir quand vous le souhaitez."

## PARTIE 5 — TOGGLE GLOBAL JEU RESPONSABLE

### Backend
Ajouter une table Settings (si pas existante) :
model Setting {
  key   String @id
  value String
}

Insérer : { key: "responsible_gaming_enabled", value: "true" }

Endpoint PATCH /admin/settings/responsible-gaming
- Body : { enabled: boolean }
- Action : met à jour la valeur en base
- Accès admin uniquement

Endpoint GET /settings/responsible-gaming
- Retourne { enabled: boolean }
- Public (appelé au chargement côté joueur)

### Frontend Admin
Dans le header du dashboard admin, ajouter un toggle visible en permanence :

┌─────────────────────────────────────┐
│  🛡️ JEU RESPONSABLE   [ON ●──  ]   │
└─────────────────────────────────────┘

- Toggle ON (vert) = module actif
- Toggle OFF (gris) = tous les popups, 
  questionnaires et triggers désactivés
- Confirmation modale avant désactivation :
  "Vous allez désactiver le module jeu 
  responsable. Confirmer ?"

### Frontend Joueur
Au chargement de chaque page joueur :
- Fetch GET /settings/responsible-gaming
- Si enabled = false → skip tous les 
  triggers automatiques (popup, questionnaire)
- Les auto-exclusions déjà actives 
  restent valides même si toggle OFF
- Les comptes bloqués par admin restent 
  bloqués même si toggle OFF
